# How-to: Connect AI Agents via the MCP Server

**When to use this:** You want to give an AI agent (Claude, Cursor, Drover Code, or any MCP-compatible client) read and write access to your SQLForge project so it can explore models, propose SQL changes, and apply them safely.

---

## What the MCP server does

SQLForge ships a built-in **Model Context Protocol (MCP)** server. When running, it exposes your project's DAG, semantic metrics, and plan/apply workflow as JSON-RPC tools that any MCP-compatible agent can call.

The server binds to a **single environment** at startup. An agent connected to `sqlforge mcp prod` can only read from and apply to `prod`.

---

## Step 1: Start the server

```bash
# From inside your project directory
sqlforge mcp prod --local
```

```
SQLForge MCP server running on http://127.0.0.1:8080
Environment: prod (schema: sqlforge__prod)
Tools registered: list_models, get_model, list_metrics, query_metric, plan_change, apply_change
```

`--local` binds to `127.0.0.1` only (default). For remote agents, omit `--local` and set `--api-key` to require authentication:

```bash
sqlforge mcp prod --api-key "your-secret-key"
```

---

## Step 2: Configure your MCP client

### Claude Desktop

Add to `~/Library/Application Support/Claude/claude_desktop_config.json`:

```json
{
  "mcpServers": {
    "sqlforge": {
      "url": "http://127.0.0.1:8080",
      "headers": {
        "Authorization": "Bearer your-secret-key"
      }
    }
  }
}
```

Restart Claude Desktop. You'll see "sqlforge" appear in the tools panel.

### Cursor

In `.cursor/mcp.json` at the project root:

```json
{
  "mcpServers": {
    "sqlforge": {
      "url": "http://127.0.0.1:8080"
    }
  }
}
```

### Drover Code

Drover Code agents discover SQLForge automatically when `sqlforge mcp` is running in the workspace. See `agents.md` for the full agent context.

## Step 3: What the agent can do

| Tool | What it does |
|------|-------------|
| `list_models` | Returns all model names, materialization types, and AST fingerprints |
| `get_model` | Returns the full SQL and column lineage for a specific model |
| `list_metrics` | Returns all semantic metrics defined in `metrics:` YAML |
| `query_metric` | Compiles a metric to dialect-specific SQL (read-only, no execution) |
| `plan_change` | Proposes new SQL for a model → returns a `plan_id` with changed/impacted model lists |
| `apply_change` | Executes a `plan_id` from `plan_change` (plans expire after 2 hours) |

### Example: ask the agent to improve a model

With Claude Desktop connected, you can prompt:

> "Look at the `customer_360` model using `get_model`. The revenue calculation looks wrong — it's summing all events, not just purchases. Fix it and apply the change."

The agent will:
1. Call `get_model("customer_360")` to read the SQL
2. Propose a corrected version via `plan_change`
3. Show you the diff and impacted models
4. Call `apply_change(plan_id)` after your approval

### Recommended agent workflow

Always prefer **plan → review → apply** rather than direct apply:

```
list_models          # discover what exists
get_model            # understand the current SQL
plan_change          # propose and review the change
apply_change         # execute after human review
```

---

## Security considerations

- **Never expose the MCP server publicly without `--api-key`.**
- The `apply_change` tool writes to your warehouse. Scope your MCP environment accordingly — prefer a non-prod environment for autonomous agent workflows.
- Plan IDs expire after 2 hours. An agent cannot apply a stale plan.
- Drover Warden `scope: sqlforge` policies apply to all SQL generated by `plan_change`.

---

## Troubleshooting

**Agent says "no tools available"**
The server must be running before the client connects. Check `http://127.0.0.1:8080/tools` in a browser — you should see a JSON list of registered tools.

**`apply_change` returns "plan not found"**
The plan expired (2-hour TTL) or the server was restarted. Run `plan_change` again to get a fresh `plan_id`.

**Agent is modifying the wrong environment**
The MCP server is bound to one environment at startup. Restart with the correct environment name.
